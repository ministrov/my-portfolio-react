# Frontend Development Guide — Portfolio Project

Принципы и конвенции для **этого** проекта. Стек: React 18 + Create React App, чистый JavaScript (без TypeScript), Framer Motion, react-i18next, react-router-dom v6, react-helmet-async, Swiper, react-icons.

> Проект на чистом JS. Там, где исходный гайд предполагал TypeScript, используем JSDoc, PropTypes (опционально) и дисциплину именования вместо системы типов.

---

## Философия разработки

- **First Principles**: SOLID, KISS (проще — лучше), DRY (не повторяйся).
- **Функциональный подход**: функциональные компоненты и хуки вместо классов; декларативный код вместо императивного.
- **Component-Driven**: приложение — композиция переиспользуемых компонентов.
- **Think Then Code**: сначала план и псевдокод, потом реализация.
- **Без избыточности типов**: TypeScript не используем сознательно. Надёжность держим на ясном именовании, маленьких функциях, PropTypes/JSDoc где это реально помогает.

---

## Архитектура и структура

### Организация проекта

- Группируй по фиче/секции, а не по типу файла, когда это удобнее для портфолио.
- Структура файла компонента:
  1. Импорты
  2. Статический контент (константы вне рендера)
  3. Экспортируемый компонент
  4. Вспомогательные подкомпоненты / функции
  5. PropTypes (если используешь)

### Конвенции именования

- **PascalCase** — компоненты (`UserProfile`), кастомные хуки-файлы можно тоже именовать ясно.
- **camelCase** — переменные, функции, методы, пропсы.
- **Префиксы**:
  - обработчики событий — `handle`: `handleClick`, `handleSubmit`
  - булевы — глагол: `isLoading`, `hasError`, `canSubmit`
  - кастомные хуки — `use`: `useScrollPosition`, `useMediaQuery`

## Типобезопасность без TypeScript

- Описывай форму пропсов через **JSDoc** над компонентом — даёт автодополнение в редакторе без сборочных затрат.
- Для критичных компонентов можно добавить **PropTypes** (отдельная зависимость `prop-types`, ставить только если реально нужно).
- Избегай «магических» строк/чисел — выноси в именованные константы (`const ANIMATION_DURATION = 0.4`).
- Проверяй null/undefined явно (guard-условия), особенно для данных, приходящих из i18n-ресурсов или внешних источников.

## React: лучшие практики

### Паттерны компонентов

- Только функциональные компоненты + хуки.
- Выноси переиспользуемую логику в кастомные хуки.
- Статический контент — в константы **вне** функции рендера.
- Обязательный cleanup в `useEffect` (отписки, таймеры, listeners).
- Все пользовательские строки — через `react-i18next` (`const { t } = useTranslation(); t('ключ')`), без хардкода текста.

### Маршрутизация (react-router v6)

- Роуты централизованно; вложенные маршруты через `<Outlet />`.
- Навигация — `<Link>` / `useNavigate`, не `window.location`.
- Ленивая загрузка страниц через `React.lazy` + `<Suspense>`.

### SEO и `<head>` (react-helmet-async)

- Оборачивай приложение в `HelmetProvider`.
- На каждой странице задавай `title`, `meta description`, при необходимости — Open Graph теги.
- Учитывай язык из i18n при формировании мета-данных.

### Производительность

- `React.memo()` — точечно, для реально перерисовывающихся компонентов.
- `useCallback` — для обработчиков, передаваемых в дочерние компоненты.
- `useMemo` — для тяжёлых вычислений.
- Избегай inline-функций и inline-объектов в JSX там, где это вызывает лишние ре-рендеры.
- Code splitting через `React.lazy` (особенно Swiper-секции и тяжёлые блоки).
- Корректные `key` в списках — стабильный id, **не индекс**.
- Анализируй бандл: `npm run analyze` (source-map-explorer).

## Обработка ошибок

### Паттерн раннего возврата

- Ошибки и крайние случаи — в начале функции.
- Early return для ошибочных условий; happy path — в конце.
- Guard-условия вместо вложенных `else`.

### Структурный подход

- Используй **Error Boundary** (классовый компонент или библиотека) для перехвата ошибок рендера, чтобы не падало всё приложение.
- Понятные пользователю сообщения об ошибках (тоже через i18n).
- Логируй ошибки для отладки.
- Для асинхронных операций — состояния `isLoading` / `hasError` и обработка `catch`.

## Состояние

- `useState` — простое локальное состояние.
- `useReducer` — сложное локальное состояние.
- **React Context** — общее состояние в пределах дерева (тема, язык, меню).
- Глобальные стейт-менеджеры (Redux/Zustand) для портфолио, как правило, **избыточны** — не добавляй без явной необходимости.

## Интернационализация (react-i18next)

- Никакого хардкода текста — всё через `t('ключ')`.
- Структурируй ключи по секциям/страницам (`hero.title`, `contact.form.email`).
- Держи файлы переводов синхронными между языками.
- Переключение языка — через `i18n.changeLanguage`.
- Учитывай язык в `<title>`/meta (react-helmet-async).

## Доступность (a11y)

- Семантический HTML.
- ARIA-атрибуты — только когда действительно нужно.
- Поддержка навигации с клавиатуры (фокус, таб-порядок), особенно для меню (hamburger-react) и слайдеров.
- Достаточный цветовой контраст.
- Логическая иерархия заголовков (`h1` → `h2` → ...).
- Понятная обратная связь об ошибках.
- `alt` для изображений; для декоративных — пустой `alt=""`.
- Уважай `prefers-reduced-motion` в анимациях Framer Motion.
